# حسابات / Hesabat — Architecture & Milestones (full detail)

Multi-tenant cloud accounting SaaS for Egyptian SMEs (`artifacts/hesabat`, route `/hesabat/`). Arabic-first (RTL, Cairo), navy+sand theme. This is the full reference; `replit.md` keeps a short summary + pointer here.

Backend routes live in `@workspace/api-server` (`routes/auth.ts`, `accounts.ts`, `dashboard.ts`, `team.ts`, `invitations.ts`, `company.ts`, `taxes.ts`, `cost-centers.ts`, `currencies.ts`, `journal.ts`, `fixed-assets.ts`, `inventory.ts`, `payroll.ts`, `bank.ts`, `advances.ts`, `customers.ts`, `suppliers.ts`, `payments.ts`, `receivables-reports.ts`); schema tables (`companies`, `users`, `sessions`, `accounts`, `invitations`, `taxes`, `cost_centers`, `currencies`, `journal_entries`, `journal_entry_lines`, `journal_entry_attachments`, `fixed_assets`, `asset_depreciation_entries`, `inventory_items`, `inventory_movements`, `employees`, `employee_pay_components`, `payroll_runs`, `payroll_run_lines`, `bank_accounts`, `bank_movements`, `bank_reconciliations`, `bank_statement_lines`, `advances`, `advance_installments`, `custodies`, `custody_attachments`, `customers`, `suppliers`, `invoices`, `invoice_lines`, `payments`, `payment_allocations`) in `@workspace/db`.

## Core architecture (read before extending)
- **Strict tenant isolation:** every business table has `company_id`; all queries scope by `req.auth.companyId` (set by `requireAuth`). Any FK that references another row (e.g. account `parentId`) MUST be re-validated to belong to the caller's company before write — a plain DB FK is not enough.
- **Native auth:** scrypt password hashing (`lib/auth.ts`), session token stored only as SHA-256 in `sessions`, httpOnly cookie (`lib/session.ts`). No vendor lock — portable to any Node+Postgres VPS.
- API client sends the cookie via `credentials: "include"` (set in `lib/api-client-react/src/custom-fetch.ts`).
- **Roles & permissions:** single source of truth is the shared `@workspace/permissions` lib (5 roles: owner/manager/accountant/data_entry/viewer; capability matrix; `hasCapability`; `ASSIGNABLE_ROLES`). The server enforces via `requireCapability(cap)` middleware (runs after `requireAuth` → 403); the frontend only hides UI. `owner` is fixed to the company creator — never invited, set, removed, or self-demoted.
- **Employee invitations:** invite-link based (no email dependency). Create returns the raw token ONCE; DB stores only its SHA-256 (`invitations` table). Accept is a PUBLIC route (`routes/invitations.ts`) — token is the credential — that creates the user + session in one tx. `users.email` is globally unique → invite-create and accept 409 on duplicate. Secret tokens ride in the URL path, so the pino-http serializer redacts `/invitations/:token` from logs (`app.ts`).
- **Journal entry numbering is concurrency-safe:** `entryNo` is allocated per company as `max(entry_no)+1`, which races under concurrent posting (e.g. two inventory movements on different items whose row locks don't conflict). EVERY transaction that allocates an entry number MUST first call `lockCompanyEntryNo(tx, companyId)` (in `lib/journal-posting.ts`) — a Postgres `pg_advisory_xact_lock(hashtext(companyId))` that serializes allocation per company and auto-releases on commit. Already wired into `createDraftJournalEntry` and both `routes/journal.ts` allocation sites (manual create + Excel import); reuse it for any future module that mints entry numbers.

## Milestones 1–6 (foundation)
- **Milestone 1:** company signup, native login/logout, financial dashboard summary, tenant-scoped chart of accounts (CRUD).
- **Milestone 2:** team members + roles + employee invitations with strict server-enforced per-company permissions.
- **Milestone 3:** company profile (name, trade name, logo, activity, tax registration, country + base currency) captured at signup, editable in a settings page (`/company`), shown on the dashboard.
- **Milestone 4 (journal entries):** bilingual accounts (`nameAr` required + `nameEn` nullable, display via `lang.startsWith("en") && nameEn ? nameEn : nameAr`), a seedable default Egyptian chart of accounts, taxes (VAT + WHT) and cost centers/projects, and double-entry journal entries with per-line multi-currency (`currency` + `exchangeRate` → `debitBase`/`creditBase`; balance checked on base amounts), file attachments (local-disk multer, forced-download), and Excel import/export (exceljs).
- **Milestone 5 (multi-currency foundation):** a company-defined `currencies` table (CRUD, `currencies:*` capabilities mirroring `taxes:*`) where each currency holds an `exchangeRate` = value of 1 unit in the company base currency (`baseAmount = amount * exchangeRate`). The journal line currency cell is a dropdown (base currency at rate 1 + active company currencies); picking a currency auto-fills the line's exchange rate (still editable), and lines with an unknown/inactive currency keep a fallback option so historic entries render. Chart of accounts auto-suggests the next account code from siblings within the parent prefix (e.g. child of `11` → `111`, `112`…; editable). Exchange rates can be auto-updated from a free external source (`open.er-api.com`, no API key) via `POST /currencies/refresh-rates` (requires `currencies:update`): it fetches `latest/{baseCurrency}` and sets each currency's `exchangeRate = 1 / rates[code]` (value of 1 unit in base), records `rateUpdatedAt`, runs the writes in one transaction, and reports updated/skipped codes; the fetch has an 8s AbortController timeout and unavailable codes are skipped. The currencies page has an "auto-update rates" button (gated by `currencies:update`) and shows the last-updated time. Manual rate edits remain available. Full multi-currency reports are a future milestone — base amounts are already persisted.
  - **UI consolidation:** the five configuration screens (company profile, team & permissions, cost centers & projects, currencies & exchange rates, taxes) are grouped under one "إعدادات الشركة / Company Settings" sidebar entry, a tabbed hub at `/settings/:tab` (`pages/settings.tsx`). The hub renders a capability-filtered tab bar and mounts each existing page component as tab content (each page's own header was de-stickied so it nests under the shell's sticky tab bar; `company` tab always shown, others gated by `team:manage`/`costCenters:read`/`currencies:read`/`taxes:read`). Invalid/unauthorized/missing tab slugs canonicalize (replace) to the first allowed tab. Legacy paths (`/company`, `/team`, `/cost-centers`, `/currencies`, `/taxes`) `Redirect` to their `/settings/<tab>`.
- **Milestone 6 (fixed assets & depreciation):** a tenant-scoped `fixed_assets` table (nameAr/nameEn/category/acquisitionDate/cost/salvageValue/usefulLifeMonths/method=`straight_line`/status=`active|disposed` + three account FKs — assetAccountId, accumulatedAccountId, expenseAccountId — all `onDelete restrict`) plus an `asset_depreciation_entries` ledger (assetId cascade, period `YYYY-MM`, amount, journalEntryId set null, **unique(assetId, period)** so a month is never double-posted). CRUD lives at `/assets` (`assets:*` capabilities mirroring `taxes:*`; the three mapped accounts are re-validated to belong to the company AND be leaf/non-group on every write). `POST /assets/depreciation/run` (body `{period:"YYYY-MM"}`, requires `assets:create`) computes straight-line monthly depreciation = `(cost − salvage) / usefulLifeMonths`, skipping assets already depreciated that month, acquired after the month, or with no remaining depreciable value (final month capped at the remainder); it emits ONE consolidated **draft** journal entry (Dr depreciation-expense / Cr accumulated-depreciation per asset) dated the last day of the period and writes the per-asset depreciation rows in the same transaction. `accumulatedDepreciation`/`netBookValue` on each listed asset are derived from the sum of its depreciation entries. The draft entry is created via the reusable `lib/journal-posting.ts` `createDraftJournalEntry(tx, opts)` helper (base-currency lines at rate 1, reuses entryNo max+1, validates balance) — shared with future payroll/inventory milestones. Frontend page `pages/fixed-assets.tsx` (route `/assets`, nav gated by `assets:read`).
- **Country-linked taxes:** the shared `@workspace/locale` catalog now covers Egypt + Gulf (countries EG/SA/AE/KW/QA/BH/OM, currencies EGP/SAR/AED/KWD/QAR/BHD/OMR/USD) and defines `TAX_KINDS` (vat/wht/income/payroll/zakat) plus per-country `TAX_TEMPLATES` (`taxTemplatesFor(country)`), each template carrying an `accountCode` from `DEFAULT_TAX_ACCOUNT_CODES` (vat→212, wht→214, payroll→215, income→216, zakat→217). At signup the chosen country's taxes are auto-seeded inside the signup transaction: `seedDefaultAccounts` (now returns a code→id Map) → `seedDefaultTaxes(tx, companyId, country, codeToId)` (in `lib/seed-taxes.ts`), each tax linked to its chart account (accounts 214–217 added to `seed-accounts.ts`, 212 renamed; `ensureTaxAccounts` lazily creates any missing tax-liability account under parent `21` for older charts). Pre-existing companies can backfill via `POST /taxes/seed-defaults` (`taxes:create`): it runs in ONE transaction that `SELECT ... FOR UPDATE` locks the company row then re-checks "already has taxes" atomically (concurrency-safe — two clicks can't double-seed), returns 409 if any tax exists, else seeds by `companies.country` and returns the `Tax[]`. Seeded taxes are normal editable/deletable rows (no schema change — `taxes.kind` is a text column). The taxes page (`pages/taxes.tsx`) has 5 kind tabs + a kind selector and a "load country defaults" button shown only when the company has zero taxes (`useSeedDefaultTaxes`). Tax *reports* are deferred to the future reports module.

## Milestone 7 — Inventory module (single warehouse)
- Tenant-scoped `inventory_items` (code unique per company, `nameAr` required + `nameEn` nullable, unit, optional category, isActive, derived `quantityOnHand`/`averageCost`, optional default `inventoryAccountId` restrict) + `inventory_movements` (item cascade, date, type `receipt|issue|adjustment`, quantity, unitCost, totalValue, `inventoryAccountId`+`counterpartAccountId` restrict, notes, `journalEntryId` set null, createdBy). Schema in `lib/db/src/schema/inventory.ts`.
- **Weighted-average cost valuation only** (no FIFO/LIFO). Receipt blends `newAvg = (curQty*curAvg + qty*unitCost)/newQty`; issue and adjustment value at the current average. Movement creation runs in ONE transaction that locks the item row `FOR UPDATE`, recomputes qty+avg, inserts the movement, and emits a DRAFT journal entry via `createDraftJournalEntry`. Negative stock is blocked (issue qty > onHand, or adjustment that would make onHand < 0); zero-qty adjustment rejected. Qty rounded 4dp, money 2dp, EPS 0.00005.
- Journal direction: receipt Dr inventory / Cr counterpart; issue Cr inventory / Dr counterpart; adjustment by sign. A draft entry is skipped only when the posting amount ≤ 0.005. All three mapped accounts (`inventoryAccountId`, `counterpartAccountId`) are re-validated to belong to the company AND be leaf (non-group) on every write (`validateAccounts`). Default inventory leaf account `114` ships in `seed-accounts.ts`.
- Capabilities: `inventory:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes in `routes/inventory.ts` (items CRUD + movements list/create), mounted in `routes/index.ts`. Frontend `pages/inventory.tsx` (route `/inventory`, nav gated by `inventory:read`): Items|Movements tabs with item create/edit/delete + record-movement modals. Out of scope (future): multiple warehouses, FIFO/LIFO, purchase/sales orders, multi-currency stock.

## Milestone 8 — Payroll & Employees module (manual amounts)
- Schema in `lib/db/src/schema/payroll.ts`: `employees` (company-scoped, code unique per company, `nameAr` required + `nameEn` nullable, jobTitle, hireDate, status `active|terminated`, baseSalary, notes), `employee_pay_components` (employee cascade, kind `allowance|deduction`, nameAr, amount, isActive), `payroll_runs` (company, **period unique per company** `YYYY-MM`, status, three account FKs salaryExpense/netPayable/deductions + totals + employeeCount + journalEntryId set null, createdBy), `payroll_run_lines` (run cascade, employeeId **restrict** + `employeeName` snapshot so historical runs survive employee edits, baseSalary/totalAllowances/totalDeductions/netPay).
- **All amounts are MANUAL** — no automatic insurance/social-security/income-tax calculation. Per active employee: gross = baseSalary + active allowances; deductions = active deductions; net = gross − deductions. A payroll run aggregates all `active` employees (terminated excluded; employees with gross ≤ EPS skipped) and emits ONE consolidated **draft** journal entry dated the last day of the period via `createDraftJournalEntry` (Dr salaryExpense=totalGross / Cr deductions=totalDeductions [only when >0] / Cr netPayable=totalNet). Money rounded 2dp, EPS 0.005.
- Guards: `unique(company, period)` blocks double-run — checked pre-tx (409) AND the `23505` unique-violation is caught post-tx and mapped to 409 (concurrency-safe, since two requests can both pass the pre-check); the deductions account is required only when total deductions > 0; per-employee negative net (deductions > gross) is rejected with 400 before posting; all three mapped accounts are re-validated to belong to the company AND be leaf (non-group). Employee update replaces its components (delete + insert) in one transaction.
- Capabilities: `payroll:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes in `routes/payroll.ts` (employees CRUD with inline `components[]`; `/payroll/runs` list/create/get-with-lines), mounted in `routes/index.ts`. **Employee update uses PATCH** (`/employees/:id`). Frontend `pages/payroll.tsx` (route `/payroll`, nav gated by `payroll:read`): Employees|Payroll Runs tabs, employee create/edit/delete modal with an allowances/deductions editor, a run modal with account pickers + month picker, and a run-detail modal showing per-employee breakdown. Out of scope (future): automatic tax/insurance calc, payslip PDFs, leave/attendance.

## Milestone 9 — Customers & Suppliers module (subsidiary ledger)
- **Subsidiary-ledger model (no stored balance):** each customer/supplier auto-gets its own leaf account created under a control GROUP account — customers under `112` (العملاء / receivables / asset), suppliers under `211` (الموردين / payables / liability). The party's balance is DERIVED on read from POSTED journal lines on its subsidiary account (`postedBalancesByAccount` sums `status='posted'` `debitBase`/`creditBase`): customer balance = debit − credit (asset), supplier = credit − debit (liability). This keeps the control account in the trial balance == sum of its subsidiaries (single source of truth, no double bookkeeping).
- Schema in `lib/db/src/schema/customers-suppliers.ts`: `customers` + `suppliers` (company cascade, code unique per company, `nameAr` required + `nameEn` nullable, type `individual|company`, taxNumber, commercialRegistration, phone, email, address, currency, creditPeriodDays, `creditLimit` **customers only**, `controlAccountId` FK restrict, `accountId` FK restrict [the auto subsidiary], isActive).
- Shared lib `artifacts/api-server/src/lib/party-ledger.ts`: `loadControlAccount` (ownership + returns isGroup/type), `generateChildAccountCode` (next suffix under control code, e.g. `112`→`112001`,`112002`; pads ≥3 digits), `postedBalancesByAccount`, `isUniqueViolation`/`isForeignKeyViolation` helpers, and `lockCompanyAccountCode` (a `pg_advisory_xact_lock(hashtext(companyId))` baked into `generateChildAccountCode` so concurrent party creates under one company can't race to the same child code — verified with an 8-way concurrent burst yielding unique sequential codes).
- Create re-validates the control account belongs to the company AND `isGroup` (400 if not), pre-checks duplicate party code (409), then in ONE tx generates the child code (under advisory lock), inserts the subsidiary leaf account (inherits the control's `type`, `parentId`=control, `isGroup`=false), and inserts the party. Update keeps the subsidiary account's `nameAr`/`nameEn` and (on control re-parent) `parentId`/`type` in sync. **Every account read is company-scoped** (joins and post-update code lookups filter `accounts.companyId`). Delete is blocked (400) if any journal lines exist on the subsidiary account (pre-check count) AND the `23503` FK-violation is caught post-tx and mapped to 400 (race-safe); on success the party and its subsidiary account are deleted together in one tx.
- Capabilities: `customers:read/create/update/delete` + `suppliers:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes `routes/customers.ts` + `routes/suppliers.ts` (mounted in `routes/index.ts`). Frontend: shared adapter-driven `components/parties/PartyManager.tsx` (one component, config + mutation wrappers passed in; customer/supplier shapes differ only by `creditLimit`) + thin `pages/customers.tsx` (route `/sales`) + `pages/suppliers.tsx` (route `/purchases`), replacing the old ComingSoon placeholders; nav gated by `customers:read` / `suppliers:read`; ar/en i18n under `customers`/`suppliers` namespaces. Out of scope (future): invoices/bills, statements, aging reports, opening balances.

## Milestone 10 — Invoicing module (line-typed invoices + payments + AR/AP reports)
- **Single `invoices` table with a `kind` discriminator** (`sales`|`purchase`) + shared `invoice_lines` (schema in `lib/db/src/schema/invoices.ts`). `invoiceNo` is unique per company+kind (separate sales/purchase sequences). Each invoice: date, dueDate, customerId|supplierId (FK restrict, exactly one set by kind), costCenterId, currency+exchangeRate, status `draft|approved|partially_paid|paid|cancelled`, notes, subtotal/discountTotal/taxTotal/total, amountPaid, journalEntryId (set null), createdBy, approvedAt.
- **Per-LINE `lineType` discriminator** (the core of this milestone): `service` (account only — revenue/expense), `inventory` (item + stock move + COGS), `fixed_asset` (registers an asset on approval). Each line: lineNo, description, accountId (FK restrict), itemId (nullable), cogsAccountId (nullable, sales-inventory only), quantity, unitPrice, discount, taxId (nullable) + taxAmount, lineTotal, costCenterId, plus asset-registration fields.
- **Approval (`POST /invoices/:id/approve`) posts ONE balanced POSTED journal entry** via `lib/journal-posting.ts`. Sales: Dr AR-control (the customer's subsidiary account) / Cr revenue + Cr tax; inventory lines additionally Cr inventory + Dr COGS at weighted-average (issue lowers stock). Purchase: Dr expense/inventory/asset + Dr tax / Cr AP-control; inventory lines raise stock (weighted-avg blend); fixed_asset lines register an asset. Stock moves reuse `lib/inventory-posting.ts` `computeMovement`/`applyStockMovement`. Cost-of-sales leaf account `514` ships in `seed-accounts.ts`.
- **Concurrency contract (critical — see also memory `hesabat-lock-ordering.md`):** approve runs entirely in ONE tx that locks the invoice row `FOR UPDATE` FIRST and reads invoice/lines/party/tax/accounts inside the tx (no stale pre-lock read), then `lockCompanyEntryNo`, then posts. Re-approve of an already-approved invoice throws `ApproveError` (400) → exactly one posted JE. Update (PATCH) and delete re-check `status='draft'` under the row lock / guarded `WHERE status='draft'` so an approve committing in between can't be overwritten or orphaned. **Global lock order = business rows (sorted) → `lockCompanyEntryNo`** in every multi-lock tx; payments lock allocated invoice rows first to match approve and avoid an approve↔payment deadlock.
- **Payments (`routes/payments.ts`):** `payments` (kind `collection`|`payment`, paymentNo, date, customer/supplier, method `cash|bank|cheque|card`, cashAccountId, amount, currency+rate, journalEntryId, createdBy) + `payment_allocations` (paymentId cascade, invoiceId restrict, amount). Create posts a cash JE (Dr cash / Cr AR for collection; Dr AP / Cr cash for payment) and applies allocations: allocations are aggregated per invoice (`allocByInvoice`), each invoice row locked `FOR UPDATE` in sorted id order, balance re-read under lock, authoritative over-allocation check, `amountPaid`+status applied exactly once (duplicate alloc lines still stored as rows but balance impact aggregated). Over-allocation / not-found / not-approved → 400.
- **AR/AP reports (`routes/receivables-reports.ts`):** party statement, AR/AP aging, outstanding/overdue, sales/purchases by party & cost-center, collections/payments over a period — all derived from posted journal lines + invoice balances (no stored balance).
- Capabilities: `invoices:read/create/update/delete` + `payments:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes mounted in `routes/index.ts`. Frontend: shared invoice editor with a per-line lineType selector + collection/payment modals + AR/AP report views; new routes `/invoices/sales` + `/invoices/purchases` (customers/suppliers keep `/sales` `/purchases` from Milestone 9), nav under an "الفواتير / Invoices" group gated by `invoices:read`; ar/en i18n. Out of scope (future): credit notes, invoice cancellation/reversal, recurring invoices, PDF rendering.

## Milestone 11 — Banks, Cash & Reconciliation module
- Schema in `lib/db/src/schema/bank.ts` (4 tables): `bank_accounts` (company cascade, nameAr req + nameEn, type `bank|cash|credit_card|loan`, bankName, accountNumber, currency, openingBalance + openingBalanceDate, `accountId` FK restrict = an existing **leaf** chart account the bank/cash account is linked to, isActive); `bank_movements` (bankAccount cascade, date, 10 `type`s, direction `in|out`, amount, currency+exchangeRate, `counterpartAccountId` FK restrict nullable, `transferAccountId` FK bank_accounts restrict nullable, `transferGroupId`, description, reference, journalEntryId set null, reconciliationId set null, isCleared, createdBy); `bank_reconciliations` (bankAccount cascade, periodStart/periodEnd, statementBalance, bookBalance, difference, status `draft|completed`, notes, adjustingEntryId set null, createdBy, completedAt); `bank_statement_lines` (reconciliation cascade, date, description, amount, direction, `matchedMovementId` set null).
- **Bank/cash account links to an EXISTING leaf chart account — it is NOT auto-created** (unlike the customers/suppliers subsidiary model). The linked account is re-validated to belong to the company AND be leaf on every write.
- **Movements post POSTED journal entries** (real cash events, like payments/invoicing — not drafts). `lib/bank-posting.ts` maps movement type → direction + balanced JE lines (Dr/Cr the bank's linked account vs the counterpart account; transfer = Dr destination bank account / Cr source). A transfer creates **TWO linked `bank_movements` rows** (shared `transferGroupId`) but **ONE** journal entry. `validateAccounts` re-checks company + leaf. Posting uses `createDraftJournalEntry` (status overridden to posted) under the standard LOCK ORDER (business rows sorted → `lockCompanyEntryNo`).
- **bookBalance = openingBalance + Σ posted movement effects up to periodEnd** (in the account currency). The bank/cash `openingBalance` is now also posted into the GL via the opening-balances module (Milestone 15) — saving opening balances sets `bank_accounts.openingBalance` AND posts a balanced opening JE line for the linked account, so the GL and bank book balance agree.
- **Reconciliation workflow:** create (computes bookBalance) → optional Excel statement upload (exceljs parse into `bank_statement_lines`) → match (toggle movements `isCleared`/`reconciliationId` via checkboxes AND link each statement line to a movement via a per-line dropdown → `statementLineMatches[].movementId` persisted to `bank_statement_lines.matchedMovementId`) → adjust (creates adjusting `bank_movements` flagged `isAdjustment=true` that post their own JEs) → complete (status `completed`, locks the period). Match validates each `statementLineMatches.movementId` belongs to THIS reconciliation's account+period (tenant isolation). Movement delete hard-deletes its JE (matches payments convention) and is **blocked if the movement OR — for transfers — EITHER linked group row is cleared/reconciled**.
- The reconciliation detail UI (`ReconciliationDetail` in `pages/bank.tsx`) shows: summary cards, the period-movements clearing table, the statement-lines table with a match dropdown, an **Outstanding items** section (uncleared movements from `detail.outstanding`, split into deposits-in-transit / outstanding-checks with a net total), and a **Reconciliation report** (book balance → outstanding → cleared/post-reconciliation balance vs statement balance → difference, plus the list of adjusting entries created, filtered by `isAdjustment`). `bank_movements.isAdjustment` (boolean, default false) is the durable marker distinguishing adjust-created movements from regular cleared ones (both otherwise share `isCleared`+`reconciliationId`).
- Capabilities: `bank:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Bank-charges expense leaf `515` ships in `seed-accounts.ts`. Routes in `routes/bank.ts` (accounts CRUD; movements list/create/delete; reconciliations list/create/get-detail/statement-upload/match/adjust/complete/report), mounted in `routes/index.ts`. Frontend `pages/bank.tsx` (route `/bank`, replaces ComingSoon, nav gated by `bank:read`): Accounts | Movements | Reconciliation tabs; ar/en i18n under `bank` namespace. Statement upload posts FormData to `/api/bank/reconciliations/:id/statement` via plain fetch. Out of scope (future): opening-balance JEs, multi-currency reconciliation, bank feeds/auto-import.

## Milestone 12 — Advances & Custodies module (payroll-tied)
- Schema in `lib/db/src/schema/payroll.ts` (4 tables): `advances` (company+employee cascade, date, amount, repaymentMonths, monthlyInstallment, startDate, endDate nullable, status `active|finished|suspended`, `advancesAccountId` FK restrict [leaf], `totalRepaid` default 0, notes, createdBy); `advance_installments` (company+advance cascade, payrollRunId set null, period `YYYY-MM`, amount; **unique(advanceId, period)** so a month's installment is never double-applied); `custodies` (company+employee cascade, type `cash|tools|devices|documents|other`, amount, receiptDate, description, status `open|settled|closed`, `settlementJournalEntryId` set null, createdBy); `custody_attachments` (company+custody cascade, fileName/objectKey/contentType/size — mirrors journal attachments, local-disk multer, forced-download).
- **Advances auto-deduct inside each payroll run** (سلف): per active employee the run sums every active advance's installment = `min(monthlyInstallment, remaining)`, gated by status=active + `startDate ≤ period end` + `remaining > EPS`. The installment is added to the employee's deductions and lowers net; the consolidated payroll draft JE credits each linked `advancesAccountId` by its aggregated installment total (Dr salaryExpense=gross / Cr deductions [regular only] / Cr advances-by-account / Cr netPayable). An `advance_installments` row is written and `totalRepaid` bumped; the advance flips to `finished` when fully repaid. Default advances leaf account `1131` (عهد الموظفين) ships in `seed-accounts.ts`.
- **Concurrency (critical):** ALL advance computation happens INSIDE the payroll `db.transaction` — it acquires `lockCompanyEntryNo(tx, companyId)` FIRST (re-entrant with `createDraftJournalEntry`'s own lock), re-reads active advances `.for("update")` so balances are fresh under the lock, recomputes installments/totals/posting lines, and bumps `totalRepaid` via **SQL arithmetic** (`totalRepaid + amount`, never an absolute precomputed value) so concurrent runs on different periods can't lost-update. Advance-independent per-employee pay (base/allowances/regular deductions/gross) is still computed pre-tx. In-tx validation failures (negative net, invalid advance account) throw a `PayrollError` caught → 400. The **deductions-account guard requires a deductions account only when `totalRegularDeductions > EPS`** (advance-only runs need no deductions account — advances credit their own account).
- **Custodies are NOT auto-deducted** (عهد) — the payroll run modal shows an open-custody alert (reuses `useListCustodies`; `custodies:read` granted to all payroll roles). **Settlement is JE-backed and reserved for the Excel-upload settle endpoint only:** `POST /custodies/:id/settle` (multipart) parses JE-shaped rows (date, docNo, debitAccount, creditAccount, description, costCenter, project, amount — accounts resolved by code, company+leaf) → `createDraftJournalEntry` → links `settlementJournalEntryId` + sets `status=settled` atomically under a guarded update. Status `settled` can **never** be set manually: both `POST /custodies` and `PATCH /custodies/:id` reject a manual transition to `settled` with 400, and the frontend custody edit dropdown hides the `settled` option unless already settled.
- Capabilities: `advances:read/create/update/delete` + `custodies:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes in `routes/advances.ts` (advances CRUD [delete blocked if installments exist]; custodies CRUD; custody settle; custody attachments add/download[forced]/delete), mounted in `routes/index.ts`. Frontend `pages/advances.tsx` (route `/advances`, replaces ComingSoon, nav gated by `advances:read`): Advances | Custodies tabs with CRUD modals, settle Excel upload (plain fetch), and an attachments detail modal; ar/en i18n under `advances` namespace. Out of scope (future): advance suspension workflow UI, custody aging reports, automated payslips.

## Milestone 14 — Excel import/export across modules + bank reconciliation completion
- **Shared Excel helper `artifacts/api-server/src/lib/excel.ts`:** multer memoryStorage upload middleware (`.xlsx` only), `cellStr`/`cellNum` cell coercers, `exportWorkbook(res, fileName, sheets)` (streams an exceljs workbook with a styled header row + auto column widths and sets the download headers), and `parseSheet` (reads the first worksheet into header-keyed row objects). **Excel routes are RAW (no openapi/codegen)** — they mirror the `journal.ts`/`customers.ts` pattern: export = `GET` streaming the workbook (frontend opens via `window.open`); import = `POST` multipart, parsed into rows, written all-or-nothing inside ONE `db.transaction` with company-scoped FK resolution (codes/names resolved to ids filtered by `companyId`, rejecting cross-tenant refs).
- **Import is master-data only; report-like modules are export-only.** Import+export: chart of accounts, customers, suppliers, inventory items, fixed assets, employees (payroll), cost centers. Export only: taxes, currencies, invoices (sales+purchase). Each import re-validates every resolved FK to the caller's company before insert and reuses the module's existing posting/locking rules (e.g. `lockCompanyEntryNo` for anything allocating an `entryNo`).
- **Shared frontend `ExcelToolbar` component** (export + import buttons with a hidden file input) + generic i18n keys under `common.*` (`exportExcel`/`importExcel`/`importing`/`importSuccess`/`importError`) so delegating module rollouts to subagents never collides on a shared translation key. The customers module is the reference implementation; other modules follow it.
- **Bank reconciliation completion (the second half of this milestone):**
  - **Account list summary:** `serializeAccounts` now attaches `latestStatementBalance` + `latestDifference` from each account's most-recent reconciliation; the Accounts table shows book/statement/difference columns + a "balanced" badge (CheckCircle2 when `|difference| < 0.005`). openapi `BankAccount` widened with both nullable fields (codegen regenerated).
  - **Structured report endpoint** `GET /bank/reconciliations/:id/report` (openapi `BankReconciliationReport`) → book balance, outstanding deposits-in-transit / outstanding-checks (with totals), adjusted statement balance, cleared book balance, statement balance, difference, isBalanced, reconciled/unreconciled statement-line counts. Plus `GET /bank/reconciliations/:id/report/export` → a formatted xlsx variance report (RAW, no codegen).
  - **CANONICAL `difference` = `statementBalance − clearedBookBalance`** (cleared = opening + only the *cleared/reconciled* movements up to periodEnd, via the `clearedBookBalanceUpTo` helper). This is the single source of truth: `buildReconciliationDetail.reconciledDifference`, the report's `difference`, the persisted `bank_reconciliations.difference`, and the account-list `latestDifference` all use it. It is persisted/refreshed on **create, match, adjust, and complete** (match/adjust changed the cleared set without previously updating the stored difference — that bug made the account list disagree with the detail). It trends to zero as the user clears items; at create nothing is cleared so it equals `statement − opening`.
  - **Report variance presentation is method-consistent:** the bank-side flow (statement → +deposits-in-transit − outstanding-checks → `adjustedStatementBalance`) is paired with the **full** book balance, and `adjustedStatementBalance − fullBookBalance == statementBalance − clearedBookBalance == difference` (algebraically identical). Never pair `adjustedStatementBalance` with `clearedBookBalance` (mixes the two reconciliation methods so the two shown numbers don't subtract to the displayed difference).

## Milestone 15 — Opening Balances module (الأرصدة الافتتاحية)
- **Single replaceable opening journal entry per company.** Users enter starting balances for ledger accounts, bank/cash accounts, customers, suppliers, and inventory items; the module posts **ONE balanced posted JE** flagged `journal_entries.isOpeningBalance=true`. Re-saving fully **replaces** the prior opening entry (delete-then-recreate inside one tx; lines cascade). A DB-level **partial unique index** `journal_entries_one_opening_per_company` `(company_id) WHERE is_opening_balance = true` guarantees at most one opening entry per company even under concurrent saves.
- **Balancing line → equity account `313` (رصيد افتتاحي).** Any imbalance between the user-entered debits/credits is absorbed by find-or-create equity account `313` (seeded in `seed-accounts.ts`; found-or-created at runtime for pre-existing companies). The JE is always balanced.
- **Schema/posting:** `isOpeningBalance` boolean on `journal_entries` (+ option in `lib/journal-posting.ts` `createDraftJournalEntry`, status overridden to posted). Built under the standard LOCK ORDER: side-effect row updates (bank + inventory) are **sorted by id** before the update loops → then `lockCompanyEntryNo` (called last inside `createDraftJournalEntry`). Routes `GET/POST /opening-balances` in `routes/opening-balances.ts`, mounted in `routes/index.ts`; openapi component is the $ref'd named input **`OpeningBalancesInput`** (NOT `<operationId>Body` — that name collides in codegen; bodies with nested arrays MUST use a differently-named $ref'd component, never inline).
- **Side-effect rows vs JE-rebuilt rows (reset semantics):** accounts/customers/suppliers are rebuilt purely from the JE, so the client omits their zeros (omission = reset). **Banks and inventory store opening state on their OWN rows** (`bank_accounts.openingBalance`, `inventory_items.quantityOnHand`/`averageCost`), so the client sends the **full snapshot incl zeros**; the server updates `openingBalance`/stock for **every** provided bank/item (including 0, to allow reset) but only pushes a JE line when `|balance| > MONEY_EPS`. Forgetting to send bank/inventory zeros leaves a stale `openingBalance` (the original reset bug).
- **Tenant isolation:** every referenced account/bank/customer/supplier/item is re-validated to the caller's company before write; a bank's linked ledger account must resolve to a company account or the save throws `BANK_NOT_FOUND`. The POST also **dedupes each input array by id (last-wins)** so a crafted payload can't post two lines to one account or double-update a row.
- Frontend `pages/opening-balances.tsx` (route `/opening-balances`, nav gated by `journal:read`, Scale icon): grouped tabs (accounts/banks/customers/suppliers/inventory) with a running balanced indicator, a date field, and save. i18n under the `openingBalances` namespace (+ `nav.openingBalances`) in ar/en. Out of scope (future): per-currency opening balances, opening-balance import via Excel.

## Milestone 16 — Multi-currency accounting engine (Phase 1.5)
Builds the full FX engine on top of the Milestone 5 currency foundation. Rate convention unchanged: **`exchangeRate` = value of 1 unit of the currency in base (EGP); `baseAmount = amount * rate`; to convert base→currency divide by rate.** All FX gains post to ONE revenue account `416` and all FX losses to ONE expense account `516` (both realized and unrealized share these). Helpers live in `artifacts/api-server/src/lib/currency.ts`: `assertAccountAcceptsCurrency`, `getRateForDate(executor, companyId, code, asOfDate, baseCurrency)` (latest dated rate `<=` date, base→1), `ensureFxAccounts(tx, companyId)` (lazy find-or-create 416/516), `recordDatedRate(...)`.
- **Account currency type (T002):** `accountsTable` gains `currencyType` (`'base'|'fixed'|'multi'`, default `'base'`) + `currency` (nullable code). `base` accepts only base-currency postings; `fixed` requires an active currency and accepts only that currency; `multi` accepts any. Enforced via `assertAccountAcceptsCurrency` in journal create/update/import, invoice approve, payments, and bank movements. Accounts route validates on write (fixed ⇒ active currency required; base ⇒ currency must be null). Accounts page exposes a currency-type selector + conditional fixed-currency picker; `onSubmit` nulls `currency` unless type is `fixed`.
- **Dated exchange rates (T003):** new `exchange_rates` table (companyId, currencyCode, rateDate, rate(18,6), source `'manual'|'auto'`, createdBy, **unique(company,currency,date)**). Manual rate edits and `refresh-rates` also upsert a dated row. `GET /currencies/rate?code=&date=` returns the latest rate `<=` date; transaction forms auto-suggest the rate by transaction date.
- **Realized FX on settlement (T004):** when a payment's rate differs from the allocated invoice's rate, the EGP difference posts to `416`/`516` so the subsidiary ledger and GL reconcile (USD invoice @50 settled @51 ⇒ diff to FX gain).
- **Currency revaluation (T005):** new `revaluations` table (**unique(companyId, asOfDate)** for idempotency) + capabilities `revaluation:read`/`revaluation:run`. `GET /revaluations` (list), `GET /revaluations/preview?asOfDate=` (per foreign-currency account: foreignBalance, baseBook, period-end rate, revaluedBase, unrealized), `POST /revaluations/run {asOfDate}` posts ONE balanced JE for the net unrealized gain/loss (409 if the date is already revalued, 400 if the period is closed). **Key incremental design:** revaluation account lines carry the foreign currency + period-end rate but **zero foreign debit/credit** — only `debitBase`/`creditBase` hold the unrealized adjustment. This deliberately breaks the `debitBase = debit*rate` invariant so `foreignBalance` stays unchanged while `baseBook` accumulates, making the next period's preview correctly incremental (no double-count). Lines are inserted manually (NOT via `createDraftJournalEntry`). Frontend `pages/revaluation.tsx` (route `/revaluation`, nav gated by `revaluation:read`, TrendingUp icon): date picker → preview table + totals, Run button gated by `revaluation:run`, past-runs table linking to each JE.
- **Multi-currency reporting (T006):** optional `reportCurrency` query param on `/reports/{trial-balance,income-statement,balance-sheet,general-ledger}`. When absent or equal to base, behavior is byte-for-byte unchanged. When a foreign currency is supplied, each report's **known base-amount fields** are converted (`round2(baseAmount / rate)`) using `getRateForDate` as of the report end date (`to`, or `asOf` for balance-sheet), and the response gains an additive `currencyInfo {baseCurrency, reportCurrency, rate}` (rate 1 when not converting). Missing rate ⇒ 400 `"لا يوجد سعر صرف لهذه العملة في هذا التاريخ"`. Display-only — stored data and posting are untouched. Export variants and party/bank/employee statements are **out of scope** (still base only). Reports page (`reportsPage` i18n namespace) shows a report-currency selector + a conversion header on the four in-scope tabs.
- **Currency-type enforcement scope:** `assertAccountAcceptsCurrency` (base/fixed/multi) is enforced on the **manual journal** surface (create + update via `validateLineRefs`, and Excel import). Document-driven postings (invoice approve, payments, bank movements) emit **base-currency** JE lines via `createDraftJournalEntry`, so the line-level currency guard does not apply there; enforcing currency-type on those flows requires document-currency-aware posting and is deliberately deferred to avoid regressing the validated base-line posting.
- Out of scope (future): per-currency opening balances, currency-type enforcement on invoice/payment/bank postings, FX on statements/exports, automatic period-end revaluation scheduling, currency on credit notes.
