## Architecture decisions

- Social platform credentials (Facebook/Instagram/LinkedIn) can be entered from the dashboard "ربط المنصات" page (connect/update/disconnect) OR set as env vars. Dashboard-entered creds are stored AES-256-GCM encrypted at rest in `social_credentials` (key derived from `CREDENTIALS_SECRET`/`SESSION_SECRET`/`ADMIN_SECRET`) — never plaintext in the DB. Resolution merges stored over env (stored wins); see `lib/social/config.ts` (`resolveFields`), `crypto.ts`, `store.ts`.
- Auto-publishing is driven by the existing DB-backed scheduler: when a scheduled social post becomes due it is marked `released` (website feed) AND dispatched to its external platform, with the per-post outcome persisted (`publishResult`/`publishError`/`platformPostId`/`publishedAt`/`publishAttempts`).
- "نشر الآن" (release) and "إعادة المحاولة" (retry) both run the same `attemptExternalPublish` dispatch; retry never throws and just re-records the outcome.
- Instagram requires a public image URL (2-step container publish). LinkedIn posts include the image via register-upload → upload-binary → reference-asset (text-only when no image); token validity is verified with a lightweight `/v2/me` call (401 ⇒ not connected, 403 tolerated).
- One-click OAuth ("Connect" buttons) lives in `lib/social/oauth.ts`. Developer *app* creds are env-only (`META_APP_ID`/`META_APP_SECRET` for Facebook+Instagram, `LINKEDIN_CLIENT_ID`/`LINKEDIN_CLIENT_SECRET`); when present the dashboard shows a Connect button (`oauthAvailable` on each connection status), otherwise it falls back to the manual key form. Flow: `GET /api/admin/social-connections/:platform/oauth-url` (adminAuth) returns the provider authorize URL carrying an HMAC-signed `state` (signed with `CREDENTIALS_SECRET`/`SESSION_SECRET`/`ADMIN_SECRET`, 10-min TTL); the provider redirects to the unauthenticated `GET /api/admin/social-connections/:platform/callback`, which verifies the signed state (proves admin-initiated), exchanges the code, enumerates the connectable targets, and persists the chosen one via the same encrypted store (never DB plaintext), then redirects back to `/admin/social-connections?connected=…`, `?select=<platform>&pending=<id>` (chooser needed), or `?social_error=…`. Meta: code → short-lived → long-lived user token → lists ALL Pages with their (non-expiring) Page tokens; Instagram additionally resolves each Page's linked `instagram_business_account` (only Pages with one are connectable). LinkedIn: code → access token → member URN via `/v2/userinfo`, plus (when `LINKEDIN_ENABLE_ORG` is truthy) the organizations the member administers via `/v2/organizationAcls` (URN + localizedName). Redirect URI is derived from `SITE_URL` (fallback `REPLIT_DOMAINS`) and must be registered in the provider app.
- Target selection (Task #6): `listOAuthTargets(platform, code)` returns an `OAuthTarget[]` (`{id, name, subtitle, fields}` — `fields` are the creds to store). The callback stores it directly when exactly one target exists; with >1 it stashes the targets in an in-memory, 10-min-TTL `pendingSelections` map (tokens NEVER persisted until chosen) and redirects to the chooser. `GET /api/admin/social-connections/:platform/pending/:id` (adminAuth) returns only `{id,name,subtitle}` (never tokens); `POST /api/admin/social-connections/:platform/select` (adminAuth, body `{pendingId,targetId}`) persists the chosen target's fields via the encrypted store and consumes the pending entry. The dashboard renders a `TargetChooser` modal when it sees `?select=&pending=`. LinkedIn org scopes (`r_organization_admin,w_organization_social`) are Marketing-Developer-Platform-gated, so they are opt-in via `LINKEDIN_ENABLE_ORG` to avoid breaking auth for unapproved apps; the LinkedIn publisher already posts with any `authorUrn` (person or organization). Each target's `fields` also carries `tokenExpiresAt` (see token expiry tracking below) when the provider returned an `expires_in`.
- Token expiry tracking: the OAuth exchange records `tokenExpiresAt` (ISO) inside the encrypted credential blob when the provider returns `expires_in` (LinkedIn ~60 days). Meta Page tokens don't expire so none is stored; manual/env tokens have no known expiry. `getTokenExpiry()` in `lib/social/config.ts` buckets it into `ok`/`expiring_soon` (≤ `EXPIRY_WARNING_DAYS`=7)/`expired`/`unknown`, surfaced on each `ConnectionStatus` (`tokenExpiresAt`/`expiryStatus`). The dashboard shows a "reconnect" nudge when expiring/expired, and the scheduler logs a warning (best-effort, never blocks dispatch) when a due post's platform token is expired/expiring. Manually re-entering an access token clears the stale `tokenExpiresAt`.

### Hesabat architecture (read before extending)
- **Strict tenant isolation:** every business table has `company_id`; all queries scope by `req.auth.companyId` (set by `requireAuth`). Any FK that references another row (e.g. account `parentId`) MUST be re-validated to belong to the caller's company before write — a plain DB FK is not enough.
- **Native auth:** scrypt password hashing (`lib/auth.ts`), session token stored only as SHA-256 in `sessions`, httpOnly cookie (`lib/session.ts`). No vendor lock — portable to any Node+Postgres VPS.
- API client sends the cookie via `credentials: "include"` (set in `lib/api-client-react/src/custom-fetch.ts`).
- **Roles & permissions:** single source of truth is the shared `@workspace/permissions` lib (5 roles: owner/manager/accountant/data_entry/viewer; capability matrix; `hasCapability`; `ASSIGNABLE_ROLES`). The server enforces via `requireCapability(cap)` middleware (runs after `requireAuth` → 403); the frontend only hides UI. `owner` is fixed to the company creator — never invited, set, removed, or self-demoted.
- **Employee invitations:** invite-link based (no email dependency). Create returns the raw token ONCE; DB stores only its SHA-256 (`invitations` table). Accept is a PUBLIC route (`routes/invitations.ts`) — token is the credential — that creates the user + session in one tx. `users.email` is globally unique → invite-create and accept 409 on duplicate. Secret tokens ride in the URL path, so the pino-http serializer redacts `/invitations/:token` from logs (`app.ts`).
- **Journal entry numbering is concurrency-safe:** `entryNo` is allocated per company as `max(entry_no)+1`, which races under concurrent posting (e.g. two inventory movements on different items whose row locks don't conflict). EVERY transaction that allocates an entry number MUST first call `lockCompanyEntryNo(tx, companyId)` (in `lib/journal-posting.ts`) — a Postgres `pg_advisory_xact_lock(hashtext(companyId))` that serializes allocation per company and auto-releases on commit. Already wired into `createDraftJournalEntry` and both `routes/journal.ts` allocation sites (manual create + Excel import); reuse it for any future module that mints entry numbers.

### Hesabat Inventory module (Milestone 7, single warehouse)
- Tenant-scoped `inventory_items` (code unique per company, `nameAr` required + `nameEn` nullable, unit, optional category, isActive, derived `quantityOnHand`/`averageCost`, optional default `inventoryAccountId` restrict) + `inventory_movements` (item cascade, date, type `receipt|issue|adjustment`, quantity, unitCost, totalValue, `inventoryAccountId`+`counterpartAccountId` restrict, notes, `journalEntryId` set null, createdBy). Schema in `lib/db/src/schema/inventory.ts`.
- **Weighted-average cost valuation only** (no FIFO/LIFO). Receipt blends `newAvg = (curQty*curAvg + qty*unitCost)/newQty`; issue and adjustment value at the current average. Movement creation runs in ONE transaction that locks the item row `FOR UPDATE`, recomputes qty+avg, inserts the movement, and emits a DRAFT journal entry via `createDraftJournalEntry`. Negative stock is blocked (issue qty > onHand, or adjustment that would make onHand < 0); zero-qty adjustment rejected. Qty rounded 4dp, money 2dp, EPS 0.00005.
- Journal direction: receipt Dr inventory / Cr counterpart; issue Cr inventory / Dr counterpart; adjustment by sign. A draft entry is skipped only when the posting amount ≤ 0.005. All three mapped accounts (`inventoryAccountId`, `counterpartAccountId`) are re-validated to belong to the company AND be leaf (non-group) on every write (`validateAccounts`). Default inventory leaf account `114` ships in `seed-accounts.ts`.
- Capabilities: `inventory:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes in `routes/inventory.ts` (items CRUD + movements list/create), mounted in `routes/index.ts`. Frontend `pages/inventory.tsx` (route `/inventory`, nav gated by `inventory:read`): Items|Movements tabs with item create/edit/delete + record-movement modals. Out of scope (future): multiple warehouses, FIFO/LIFO, purchase/sales orders, multi-currency stock.

### Hesabat Payroll & Employees module (Milestone 8, manual amounts)
- Schema in `lib/db/src/schema/payroll.ts`: `employees` (company-scoped, code unique per company, `nameAr` required + `nameEn` nullable, jobTitle, hireDate, status `active|terminated`, baseSalary, notes), `employee_pay_components` (employee cascade, kind `allowance|deduction`, nameAr, amount, isActive), `payroll_runs` (company, **period unique per company** `YYYY-MM`, status, three account FKs salaryExpense/netPayable/deductions + totals + employeeCount + journalEntryId set null, createdBy), `payroll_run_lines` (run cascade, employeeId **restrict** + `employeeName` snapshot so historical runs survive employee edits, baseSalary/totalAllowances/totalDeductions/netPay).
- **All amounts are MANUAL** — no automatic insurance/social-security/income-tax calculation. Per active employee: gross = baseSalary + active allowances; deductions = active deductions; net = gross − deductions. A payroll run aggregates all `active` employees (terminated excluded; employees with gross ≤ EPS skipped) and emits ONE consolidated **draft** journal entry dated the last day of the period via `createDraftJournalEntry` (Dr salaryExpense=totalGross / Cr deductions=totalDeductions [only when >0] / Cr netPayable=totalNet). Money rounded 2dp, EPS 0.005.
- Guards: `unique(company, period)` blocks double-run — checked pre-tx (409) AND the `23505` unique-violation is caught post-tx and mapped to 409 (concurrency-safe, since two requests can both pass the pre-check); the deductions account is required only when total deductions > 0; per-employee negative net (deductions > gross) is rejected with 400 before posting; all three mapped accounts are re-validated to belong to the company AND be leaf (non-group). Employee update replaces its components (delete + insert) in one transaction.
- Capabilities: `payroll:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes in `routes/payroll.ts` (employees CRUD with inline `components[]`; `/payroll/runs` list/create/get-with-lines), mounted in `routes/index.ts`. **Employee update uses PATCH** (`/employees/:id`). Frontend `pages/payroll.tsx` (route `/payroll`, nav gated by `payroll:read`): Employees|Payroll Runs tabs, employee create/edit/delete modal with an allowances/deductions editor, a run modal with account pickers + month picker, and a run-detail modal showing per-employee breakdown. Out of scope (future): automatic tax/insurance calc, payslip PDFs, leave/attendance.

### Hesabat Customers & Suppliers module (Milestone 9, subsidiary ledger)
- **Subsidiary-ledger model (no stored balance):** each customer/supplier auto-gets its own leaf account created under a control GROUP account — customers under `112` (العملاء / receivables / asset), suppliers under `211` (الموردين / payables / liability). The party's balance is DERIVED on read from POSTED journal lines on its subsidiary account (`postedBalancesByAccount` sums `status='posted'` `debitBase`/`creditBase`): customer balance = debit − credit (asset), supplier = credit − debit (liability). This keeps the control account in the trial balance == sum of its subsidiaries (single source of truth, no double bookkeeping).
- Schema in `lib/db/src/schema/customers-suppliers.ts`: `customers` + `suppliers` (company cascade, code unique per company, `nameAr` required + `nameEn` nullable, type `individual|company`, taxNumber, commercialRegistration, phone, email, address, currency, creditPeriodDays, `creditLimit` **customers only**, `controlAccountId` FK restrict, `accountId` FK restrict [the auto subsidiary], isActive).
- Shared lib `artifacts/api-server/src/lib/party-ledger.ts`: `loadControlAccount` (ownership + returns isGroup/type), `generateChildAccountCode` (next suffix under control code, e.g. `112`→`112001`,`112002`; pads ≥3 digits), `postedBalancesByAccount`, `isUniqueViolation`/`isForeignKeyViolation` helpers, and `lockCompanyAccountCode` (a `pg_advisory_xact_lock(hashtext(companyId))` baked into `generateChildAccountCode` so concurrent party creates under one company can't race to the same child code — verified with an 8-way concurrent burst yielding unique sequential codes).
- Create re-validates the control account belongs to the company AND `isGroup` (400 if not), pre-checks duplicate party code (409), then in ONE tx generates the child code (under advisory lock), inserts the subsidiary leaf account (inherits the control's `type`, `parentId`=control, `isGroup`=false), and inserts the party. Update keeps the subsidiary account's `nameAr`/`nameEn` and (on control re-parent) `parentId`/`type` in sync. **Every account read is company-scoped** (joins and post-update code lookups filter `accounts.companyId`). Delete is blocked (400) if any journal lines exist on the subsidiary account (pre-check count) AND the `23503` FK-violation is caught post-tx and mapped to 400 (race-safe); on success the party and its subsidiary account are deleted together in one tx.
- Capabilities: `customers:read/create/update/delete` + `suppliers:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes `routes/customers.ts` + `routes/suppliers.ts` (mounted in `routes/index.ts`). Frontend: shared adapter-driven `components/parties/PartyManager.tsx` (one component, config + mutation wrappers passed in; customer/supplier shapes differ only by `creditLimit`) + thin `pages/customers.tsx` (route `/sales`) + `pages/suppliers.tsx` (route `/purchases`), replacing the old ComingSoon placeholders; nav gated by `customers:read` / `suppliers:read`; ar/en i18n under `customers`/`suppliers` namespaces. Out of scope (future): invoices/bills, statements, aging reports, opening balances.

### Hesabat Invoicing module (Milestone 10, line-typed invoices + payments + AR/AP reports)
- **Single `invoices` table with a `kind` discriminator** (`sales`|`purchase`) + shared `invoice_lines` (schema in `lib/db/src/schema/invoices.ts`). `invoiceNo` is unique per company+kind (separate sales/purchase sequences). Each invoice: date, dueDate, customerId|supplierId (FK restrict, exactly one set by kind), costCenterId, currency+exchangeRate, status `draft|approved|partially_paid|paid|cancelled`, notes, subtotal/discountTotal/taxTotal/total, amountPaid, journalEntryId (set null), createdBy, approvedAt.
- **Per-LINE `lineType` discriminator** (the core of this milestone): `service` (account only — revenue/expense), `inventory` (item + stock move + COGS), `fixed_asset` (registers an asset on approval). Each line: lineNo, description, accountId (FK restrict), itemId (nullable), cogsAccountId (nullable, sales-inventory only), quantity, unitPrice, discount, taxId (nullable) + taxAmount, lineTotal, costCenterId, plus asset-registration fields.
- **Approval (`POST /invoices/:id/approve`) posts ONE balanced POSTED journal entry** via `lib/journal-posting.ts`. Sales: Dr AR-control (the customer's subsidiary account) / Cr revenue + Cr tax; inventory lines additionally Cr inventory + Dr COGS at weighted-average (issue lowers stock). Purchase: Dr expense/inventory/asset + Dr tax / Cr AP-control; inventory lines raise stock (weighted-avg blend); fixed_asset lines register an asset. Stock moves reuse `lib/inventory-posting.ts` `computeMovement`/`applyStockMovement`. Cost-of-sales leaf account `514` ships in `seed-accounts.ts`.
- **Concurrency contract (critical — see also memory `hesabat-lock-ordering.md`):** approve runs entirely in ONE tx that locks the invoice row `FOR UPDATE` FIRST and reads invoice/lines/party/tax/accounts inside the tx (no stale pre-lock read), then `lockCompanyEntryNo`, then posts. Re-approve of an already-approved invoice throws `ApproveError` (400) → exactly one posted JE. Update (PATCH) and delete re-check `status='draft'` under the row lock / guarded `WHERE status='draft'` so an approve committing in between can't be overwritten or orphaned. **Global lock order = business rows (sorted) → `lockCompanyEntryNo`** in every multi-lock tx; payments lock allocated invoice rows first to match approve and avoid an approve↔payment deadlock.
- **Payments (`routes/payments.ts`):** `payments` (kind `collection`|`payment`, paymentNo, date, customer/supplier, method `cash|bank|cheque|card`, cashAccountId, amount, currency+rate, journalEntryId, createdBy) + `payment_allocations` (paymentId cascade, invoiceId restrict, amount). Create posts a cash JE (Dr cash / Cr AR for collection; Dr AP / Cr cash for payment) and applies allocations: allocations are aggregated per invoice (`allocByInvoice`), each invoice row locked `FOR UPDATE` in sorted id order, balance re-read under lock, authoritative over-allocation check, `amountPaid`+status applied exactly once (duplicate alloc lines still stored as rows but balance impact aggregated). Over-allocation / not-found / not-approved → 400.
- **AR/AP reports (`routes/receivables-reports.ts`):** party statement, AR/AP aging, outstanding/overdue, sales/purchases by party & cost-center, collections/payments over a period — all derived from posted journal lines + invoice balances (no stored balance).
- Capabilities: `invoices:read/create/update/delete` + `payments:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes mounted in `routes/index.ts`. Frontend: shared invoice editor with a per-line lineType selector + collection/payment modals + AR/AP report views; new routes `/invoices/sales` + `/invoices/purchases` (customers/suppliers keep `/sales` `/purchases` from Milestone 9), nav under an "الفواتير / Invoices" group gated by `invoices:read`; ar/en i18n. Out of scope (future): credit notes, invoice cancellation/reversal, recurring invoices, PDF rendering.

### Hesabat Banks, Cash & Reconciliation module (Milestone 11)
- Schema in `lib/db/src/schema/bank.ts` (4 tables): `bank_accounts` (company cascade, nameAr req + nameEn, type `bank|cash|credit_card|loan`, bankName, accountNumber, currency, openingBalance + openingBalanceDate, `accountId` FK restrict = an existing **leaf** chart account the bank/cash account is linked to, isActive); `bank_movements` (bankAccount cascade, date, 10 `type`s, direction `in|out`, amount, currency+exchangeRate, `counterpartAccountId` FK restrict nullable, `transferAccountId` FK bank_accounts restrict nullable, `transferGroupId`, description, reference, journalEntryId set null, reconciliationId set null, isCleared, createdBy); `bank_reconciliations` (bankAccount cascade, periodStart/periodEnd, statementBalance, bookBalance, difference, status `draft|completed`, notes, adjustingEntryId set null, createdBy, completedAt); `bank_statement_lines` (reconciliation cascade, date, description, amount, direction, `matchedMovementId` set null).
- **Bank/cash account links to an EXISTING leaf chart account — it is NOT auto-created** (unlike the customers/suppliers subsidiary model). The linked account is re-validated to belong to the company AND be leaf on every write.
- **Movements post POSTED journal entries** (real cash events, like payments/invoicing — not drafts). `lib/bank-posting.ts` maps movement type → direction + balanced JE lines (Dr/Cr the bank's linked account vs the counterpart account; transfer = Dr destination bank account / Cr source). A transfer creates **TWO linked `bank_movements` rows** (shared `transferGroupId`) but **ONE** journal entry. `validateAccounts` re-checks company + leaf. Posting uses `createDraftJournalEntry` (status overridden to posted) under the standard LOCK ORDER (business rows sorted → `lockCompanyEntryNo`).
- **bookBalance = openingBalance + Σ posted movement effects up to periodEnd** (in the account currency). **Opening balance is NOT posted as an opening JE** (the GL reflects movements only) — known limitation; opening balances are a deferred milestone.
- **Reconciliation workflow:** create (computes bookBalance) → optional Excel statement upload (exceljs parse into `bank_statement_lines`) → match (toggle movements `isCleared`/`reconciliationId`; statement lines link to movements) → adjust (creates adjusting `bank_movements` that post their own JEs) → complete (status `completed`, locks the period). Match validates each `statementLineMatches.movementId` belongs to THIS reconciliation's account+period (tenant isolation). Movement delete hard-deletes its JE (matches payments convention) and is **blocked if the movement OR — for transfers — EITHER linked group row is cleared/reconciled**.
- Capabilities: `bank:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Bank-charges expense leaf `515` ships in `seed-accounts.ts`. Routes in `routes/bank.ts` (accounts CRUD; movements list/create/delete; reconciliations list/create/get-detail/statement-upload/match/adjust/complete/report), mounted in `routes/index.ts`. Frontend `pages/bank.tsx` (route `/bank`, replaces ComingSoon, nav gated by `bank:read`): Accounts | Movements | Reconciliation tabs; ar/en i18n under `bank` namespace. Statement upload posts FormData to `/api/bank/reconciliations/:id/statement` via plain fetch. Out of scope (future): opening-balance JEs, multi-currency reconciliation, bank feeds/auto-import.
