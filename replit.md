Flow: `GET /api/admin/social-connections/:platform/oauth-url` (adminAuth) returns the provider authorize URL carrying an HMAC-signed `state` (signed with `CREDENTIALS_SECRET`/`SESSION_SECRET`/`ADMIN_SECRET`, 10-min TTL); the provider redirects to the unauthenticated `GET /api/admin/social-connections/:platform/callback`, which verifies the signed state (proves admin-initiated), exchanges the code, enumerates the connectable targets, and persists the chosen one via the same encrypted store (never DB plaintext), then redirects back to `/admin/social-connections?connected=‚Ä¶`, `?select=<platform>&pending=<id>` (chooser needed), or `?social_error=‚Ä¶`. Redirect URI is derived from `SITE_URL` (fallback `REPLIT_DOMAINS`) and must be registered in the provider app.

The dashboard renders a `TargetChooser` modal when it sees `?select=&pending=`. LinkedIn org scopes (`r_organization_admin,w_organization_social`) are Marketing-Developer-Platform-gated, so they are opt-in via `LINKEDIN_ENABLE_ORG` to avoid breaking auth for unapproved apps; the LinkedIn publisher already posts with any `authorUrn` (person or organization). Each target's `fields` also carries `tokenExpiresAt` (see token expiry tracking below) when the provider returned an `expires_in`.
- Token expiry tracking: the OAuth exchange records `tokenExpiresAt` (ISO) inside the encrypted credential blob when the provider returns `expires_in` (LinkedIn ~60 days). Meta Page tokens don't expire so none is stored; manual/env tokens have no known expiry. `getTokenExpiry()` in `lib/social/config.ts` buckets it into `ok`/`expiring_soon` (‚â§ `EXPIRY_WARNING_DAYS`=7)/`expired`/`unknown`, surfaced on each `ConnectionStatus` (`tokenExpiresAt`/`expiryStatus`). Manually re-entering an access token clears the stale `tokenExpiresAt`.

### Hesabat architecture (read before extending)
- **Strict tenant isolation:** every business table has `company_id`; all queries scope by `req.auth.companyId` (set by `requireAuth`). Any FK that references another row (e.g. account `parentId`) MUST be re-validated to belong to the caller's company before write ‚Äî a plain DB FK is not enough.
- **Native auth:** scrypt password hashing (`lib/auth.ts`), session token stored only as SHA-256 in `sessions`, httpOnly cookie (`lib/session.ts`). No vendor lock ‚Äî portable to any Node+Postgres VPS.
- API client sends the cookie via `credentials: "include"` (set in `lib/api-client-react/src/custom-fetch.ts`).
- **Roles & permissions:** single source of truth is the shared `@workspace/permissions` lib (5 roles: owner/manager/accountant/data_entry/viewer; capability matrix; `hasCapability`; `ASSIGNABLE_ROLES`). The server enforces via `requireCapability(cap)` middleware (runs after `requireAuth` ‚Üí 403); the frontend only hides UI. `owner` is fixed to the company creator ‚Äî never invited, set, removed, or self-demoted.
- **Employee invitations:** invite-link based (no email dependency). Create returns the raw token ONCE; DB stores only its SHA-256 (`invitations` table). Accept is a PUBLIC route (`routes/invitations.ts`) ‚Äî token is the credential ‚Äî that creates the user + session in one tx. `users.email` is globally unique ‚Üí invite-create and accept 409 on duplicate. Secret tokens ride in the URL path, so the pino-http serializer redacts `/invitations/:token` from logs (`app.ts`).
- **Journal entry numbering is concurrency-safe:** `entryNo` is allocated per company as `max(entry_no)+1`, which races under concurrent posting (e.g. two inventory movements on different items whose row locks don't conflict). EVERY transaction that allocates an entry number MUST first call `lockCompanyEntryNo(tx, companyId)` (in `lib/journal-posting.ts`) ‚Äî a Postgres `pg_advisory_xact_lock(hashtext(companyId))` that serializes allocation per company and auto-releases on commit. Already wired into `createDraftJournalEntry` and both `routes/journal.ts` allocation sites (manual create + Excel import); reuse it for any future module that mints entry numbers.

### Hesabat Inventory module (Milestone 7, single warehouse)
- Tenant-scoped `inventory_items` (code unique per company, `nameAr` required + `nameEn` nullable, unit, optional category, isActive, derived `quantityOnHand`/`averageCost`, optional default `inventoryAccountId` restrict) + `inventory_movements` (item cascade, date, type `receipt|issue|adjustment`, quantity, unitCost, totalValue, `inventoryAccountId`+`counterpartAccountId` restrict, notes, `journalEntryId` set null, createdBy). Schema in `lib/db/src/schema/inventory.ts`.
- **Weighted-average cost valuation only** (no FIFO/LIFO). Receipt blends `newAvg = (curQty*curAvg + qty*unitCost)/newQty`; issue and adjustment value at the current average. Movement creation runs in ONE transaction that locks the item row `FOR UPDATE`, recomputes qty+avg, inserts the movement, and emits a DRAFT journal entry via `createDraftJournalEntry`. Negative stock is blocked (issue qty > onHand, or adjustment that would make onHand < 0); zero-qty adjustment rejected. Qty rounded 4dp, money 2dp, EPS 0.00005.
- Journal direction: receipt Dr inventory / Cr counterpart; issue Cr inventory / Dr counterpart; adjustment by sign. A draft entry is skipped only when the posting amount ‚â§ 0.005. All three mapped accounts (`inventoryAccountId`, `counterpartAccountId`) are re-validated to belong to the company AND be leaf (non-group) on every write (`validateAccounts`). Default inventory leaf account `114` ships in `seed-accounts.ts`.
- Capabilities: `inventory:read/create/update/delete` in `@workspace/permissions` (owner/manager full; accountant r/c/u; data_entry r/c; viewer r). Routes in `routes/inventory.ts` (items CRUD + movements list/create), mounted in `routes/index.ts`. Frontend `pages/inventory.tsx` (route `/inventory`, nav gated by `inventory:read`): Items|Movements tabs with item create/edit/delete + record-movement modals. Out of scope (future): multiple warehouses, FIFO/LIFO, purchase/sales orders, multi-currency stock.
